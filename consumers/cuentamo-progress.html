En una línea

  • Banco: listo. Los movimientos entregan la forma canónica completa — account estructurado, balanceAfter, valueDate, importe decimal con signo y contraparte normalizada.
  • Bróker: schema investment@2 en producción, con discriminador trade / cash. Trade Republic ya lo emite.
  • Datos preexistentes: se convierten solos. Una migración al arrancar re-normaliza el histórico; el siguiente Sync reenvía lo corregido a los sinks re-escribibles.
  • Podéis empezar la ingesta ya. La forma canónica e investment@2 son estables.

A

Banco — canónico (sink.normalize)

Con sink.normalize activo, un consumidor recibe la forma uniforme para todas las fuentes. El campo account, antes un string, es ahora un objeto:

{
  "id": "…", "date": "2026-03-01", "amount": -12.50, "currency": "EUR",
  "direction": "debit", "description": "…", "counterparty": "…",
  "category": "…", "type": "…", "source": "ing-es",
  "account": { "iban": "ES…", "last4": "1332", "groupId": "…", "currency": "EUR" },
  "valueDate": "2026-03-02",     // solo si la fuente lo captura
  "balanceAfter": 1234.56,       // solo si la fuente lo captura
  "extra": { … }                 // todos los campos raw de la fuente
}
  • date es la fecha contable → mapeadla a vuestro bookedDate.
  • account.last4 es el número que el usuario reconoce (últimos 4 de la tarjeta/cuenta), no un id interno.
  • amount siempre decimal-mayor con signo; direction = debit/credit.

B

Bróker — investment@2

Cada registro es una operación, discriminada por recordType. Los campos opcionales se omiten cuando no existen (no van a null); un side/kind no reconocido se conserva verbatim.

Operación de valores — trade

{
  "recordType": "trade", "date": "…", "currency": "EUR",
  "side": "buy | sell | dividend | split | transfer_in | transfer_out",
  "instrument": { "isin": "…", "ticker": "…", "mic": "…", "name": "…", "assetClass": "…" },
  "units": 10, "price": 25.5,
  "grossAmount": 255, "commission": 1.2, "taxWithheld": 0, "netAmount": 256.2,
  "exchangeRate": 1, "settlementAccount": "…"
}

Movimiento de efectivo — cash

{
  "recordType": "cash", "date": "…", "currency": "EUR",
  "kind": "interest | deposit | withdrawal | fee | tax | other",
  "amount": 4.20, "direction": "credit", "description": "…", "account": "…"
}

Trade Republic Listo emite ya investment@2: recordType inferido por ISIN, side/kind desde el eventType, instrument{isin,name} y settlementAccount.

Pendiente units/price/commission/grossAmount/taxWithheld viven en extra.detail como texto localizado ("1,47 × 8,13 €"): se conservan, pero aún no se parsean a campos estructurados.

C

Conversión de datos ya existentes

  • Migración única al arrancar — re-normaliza en el sitio el histórico ya guardado: los bancos ganan balanceAfter/valueDate; los registros de Trade Republic guardados como transaction@1 se promueven a investment@2. Idempotente y sellada para correr una sola vez.
  • Sello por registro (srcVersion) — cada registro guarda la versión del source con la que se procesó por última vez. Es metadato del store; no viaja en el record que os llega.
  • Reenvío a sinks — tras convertir, se resetea el ledger de entrega solo de los sinks read/write (carpeta / Drive / Dropbox / WebDAV / S3) → el siguiente Sync reescribe el manifiesto con la forma nueva. Los sinks de una vía (descarga, HTTP push) no se tocan.
D

Qué podéis hacer ya en Cuéntamo

  1. Implementar una ingesta contra la forma canónica (banco) + investment@2 (bróker) — misma forma para todas las fuentes.
  2. Idempotencia por internalId (Habeas ya deduplica, pero toleradlo por si hay reenvíos tras la migración).
  3. Mapear date → bookedDate, account.last4/account.iban a vuestra identidad de cuenta, y direction al signo.
E

Pendiente conocido no bloquea

  • Trade Republic: parseo estructurado de units/price/commission desde extra.detail.
  • valueDate/balanceAfter no existen en algunos feeds (WiZink / FECI / CaixaBank) — es esperado, no un fallo.
  • Más fuentes de bróker (investment@2) por grabación.
